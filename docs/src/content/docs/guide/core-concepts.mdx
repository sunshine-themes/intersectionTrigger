## The IntersectionObserver API

The [IntersectionObserver API](https://developer.mozilla.org/en-US/docs/Web/API/IntersectionObserver) provides an asynchronous way to observe changes in the intersection of a target element with an ancestor element or the viewport. It is the modern, performant alternative to tracking element visibility.

## How IntersectionTrigger Enhances It

The native `IntersectionObserver` has a significant limitation: when a **target element is taller than the root element**, the observer fires its callback only once -- when the target first enters the root. It cannot detect when the target finishes crossing through the root boundary.

IntersectionTrigger resolves this by layering its own position-tracking logic on top of the native observer. It calculates precise **enter** and **leave** positions for both the trigger element and the root boundary, then uses those positions to emit directional callbacks (`onEnter`, `onLeave`, `onEnterBack`, `onLeaveBack`).

This means you get reliable, granular intersection events regardless of the relative sizes of your elements.

## Intersection vs Scroll Events

A common approach to visibility detection is listening for `scroll` events and computing element positions manually. This has several drawbacks:

- **Scroll events run on the main thread.** Every scroll handler blocks rendering and can cause jank, especially when multiple listeners are attached or when the handler performs DOM reads and writes.
- **Frequent layout thrashing.** Reading `getBoundingClientRect()` inside a scroll handler forces the browser to recalculate layout on every invocation.

IntersectionTrigger avoids these problems by relying on `IntersectionObserver`, which handles observation **off the main thread**. The browser itself manages the intersection check at an optimal frequency. IntersectionTrigger only adds lightweight supplementary logic (via `requestAnimationFrame`) for cases where the native observer alone is insufficient.

## Architecture: Core + Plugins

IntersectionTrigger is built with a modular architecture:

- **Core** (`IntersectionTrigger` class) handles intersection detection, trigger management, position calculations, and lifecycle control.
- **Plugins** extend the core with additional capabilities:
  - **Animation** -- controlled and scroll-linked animations via AnimeJS.
  - **Toggle Class** -- CSS class toggling on intersection events.
  - **Guides** -- visual debugging overlays for trigger and root positions.

Plugins are opt-in. You register only the ones you need via `IntersectionTrigger.registerPlugins()`, keeping bundle size minimal.

## Instance Lifecycle

An IntersectionTrigger instance follows this lifecycle:

1. **Create** -- Instantiate with `new IntersectionTrigger(options)`. The constructor merges your options with defaults, resolves the root element, parses position values, and creates the internal `IntersectionObserver`.
2. **Add triggers** -- Call `add(trigger, options?)` to register one or more elements. Each trigger gets its own position data and state tracking.
3. **Observe** -- The observer watches your triggers automatically. IntersectionTrigger computes enter/leave thresholds and begins firing callbacks as elements cross the configured boundaries.
4. **Callbacks** -- Your `onEnter`, `onLeave`, `onEnterBack`, and `onLeaveBack` handlers fire in response to directional intersection changes.
5. **Update / Kill** -- Call `update()` to recalculate positions (e.g., after layout changes). Call `kill()` to destroy the observer, remove listeners, and clean up all state.

```js
import IntersectionTrigger from 'intersectiontrigger';

// 1. Create
const it = new IntersectionTrigger({ root: '#scroll-container' });

// 2. Add triggers
it.add('.element', {
	onEnter: (trigger) => console.log('Entered:', trigger),
	onLeave: (trigger) => console.log('Left:', trigger),
});

// 3-4. Observation and callbacks happen automatically

// 5. Clean up when done
it.kill();
```
